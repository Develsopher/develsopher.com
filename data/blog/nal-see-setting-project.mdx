## 협업을 위한 준비과정

우리 팀의 프론트엔드 담당자는 총 3명이다.(나 포함)

물론 많은 인원이 아니기에 의사결정이 빨랐고 협업을 위한 세팅은 어렵지 않고 빠르게 진행할 수 있었다.

### 1. Github 브랜치 전략 및 Jira 이슈 관리

우선 우리는 다양한 Github 브랜치 전략중 `Git Flow`[참고](https://nvie.com/posts/a-successful-git-branching-model/)를 선택했다. Git Flow방식을 선택한 이유는 과거의 프로젝트에서도 사용한 경험이 있다는 익숙함이였다. 하지만, Gif FLow 방식은 소규모 팀 그리고 웹 어플리케이션은 특성상 사용자는 항상 최신의 단일 버전만 사용한다는 점에서 적절하지 않을 수 있다. 우리는 일단 `hotfix`, `release`와 같은 부분을 제외하기로 하였다.

조금 더 간편하게 관리할 수 있는 전략인 `Github Flow`도 고려하고 있다.

또한, 이슈 관리를 위해 Jira를 사용하기로 하였다. Jira는 이슈 관리뿐만 아니라 스프린트 관리, 일정 관리 등 다양한 기능을 제공하기 때문에 우리의 프로젝트에 적합하다고 판단하였다. 일주일 단위로 스프린트 하고 `팀 Slack`을 연동하여 이슈가 생성되면 슬랙으로 알림이 가도록 설정하였다.

Github와 연동이 쉬워 Jira의 이슈카드에 커밋이력을 확인할 수 있다는 점이 가장 큰 장점이라고 생각한다.
대신 컨벤션을 지키기로 하였는데 `feature/NS-기능명`으로 브랜치를 생성하기로 하였다. 그리고 커밋 메시지에 이력이 트래킹 될 수 있도록 이슈번호를 반드시 적기로하였다.

### 2. 디자인 시스템 만들기

세 가지 주요 디자인 및 스타일링 솔루션인 `Ant Design Mobile`, `Tailwind CSS`, 그리고 `Shadcn UI`를 통합하여 강력하고 일관된 디자인 시스템을 구축하기로 결정했다.

Tailwind css를 통해 가장 로우레벨의 스타일링을 직접 구현하기로 하였다. 컴포넌트 내에서 직접 스타일을 적용함으로써 세밀한 디자인 조정을 신속하게 수행할 예정이다.

그리고 환상의 짝궁 Shadcn UI는 필요한 공용컴포넌트의 틀을 미리 만들때 정말 활용도가 높다. 우리의 프로젝트의 특성에 맞춰 필요한 요소들을 커스터마이징 할 예정이다.

마지막으로 Ant Design Mobile은 모바일 뷰에 최적화된 컴포넌트를 제공해주는데 웬만하면 직접 구현을 하겠지만 snap별로 Drawer의 높이가 달라진다는 등 직접 구현하기어려운 부분에 적극 활용하기로 하였다.

### 3. 폴더 구조 세팅

> 뭐 이런거 까지 미리 다 세팅하고 가야돼?

이런생각이 들 수 있다. 나도 얼마전까지는 그랬다. 하지만 프로젝트가 커지고, 협업이 필요해지면서 이런것들이 필요하다는 것을 깨달았다.

개인 프로젝트를 진행하더라도 그때 그때 기분과 감정(?)에 따라 폴더구조가 뒤죽박죽이 되는 경우가 많았다.
에디터에서 뒤죽박죽 섞인 파일들을 보면 심미적으로도 좋지는 않다.

팀 규칙없는 폴더 구조는 작업의 엄청난 비효율성을 야기한다. 이미 만든 컴포넌트인데 이름만 바껴서 똑같은 걸 만들고 있는 경우가 많고 각자의 스타일대로 폴더구조를 만들다보면 서로가 서로의 코드에 접근성이 떨어지고 관심이 떨어지는것 같다.

참조한 영상은 다음과 같다. [Youtube Link](https://www.youtube.com/watch?v=UUga4-z7b6s)

![폴더구조 참조](https://develsopher-nextjs.s3.ap-northeast-2.amazonaws.com/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA+2024-03-21+%E1%84%8B%E1%85%A9%E1%84%92%E1%85%AE+1.29.29.png)

- assets: 번들링과정에 참여하는 이미지, 폰트, 아이콘 등의 파일들을 저장하는 폴더
- components: 공용 컴포넌트들을 저장하는 폴더
- features: 페이지대신 각각의 기능별의 폴더를 구성하였다.
- hooks: 커스텀 훅을 저장하는 폴더
- lib: 공용 함수, 유틸리티 함수를 저장하는 폴더
- mocks: API 연결전 프론트레벨에서 mocking server 구성하는 폴더
- routes: 라우팅을 위한 폴더
- services: 데이터를 페칭하기 위한 api 함수 또는 react-query 에필요한 함수를 저장하는 폴더
- stores: 전역 상태를 관리하기 위한 폴더
- types: 타입스크립트 타입을 저장하는 폴더

#### 공용이냐 vs 로컬이냐

`feature`기준으로 생각을 하였을땐 두군데 이상에서의 feature에 사용될 경우 src 이하 공용폴더에 위치하게 구성하였다. 만약 해당 feature에서만 사용되는 경우에는 해당 feature폴더 내부에 위치하게 구성하였다.

![feature 이하 폴더구조](https://develsopher-nextjs.s3.ap-northeast-2.amazonaws.com/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA+2024-03-21+%E1%84%8B%E1%85%A9%E1%84%92%E1%85%AE+1.33.43.png)

next js 에서 폴더구조가 router에 영향을 미치기 때문에 더더욱 중요한데 해당 프로젝트에서 흔히 쓰이는 전략을 사용하였다.

### 결론

사실 협업을 위한 프로젝트 세팅에는 특별한 답이 존재하는것은 아니다. 그리고 어느정도까지 미리 고려하고 세팅해야하는것도 정해지지 않았다. 프로젝트에 필요한 효율적인 세팅이 무엇인지 깨닫는 방법은 최대한 많은 프로젝트를 만들고 협업을 경험하는거라고 생각한다.
